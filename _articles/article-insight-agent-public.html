1. Introduction — What Problem Are We Solving?

Analysts spend hours drilling through dashboards. They want to type "which of my SKUs drove the OOS spike last week?" and get an answer — not click through 5 filters.

Key points

  • We built an AI assistant embedded inside a retail analytics platform for FMCG / CPG companies.
  • Users are category managers, trade marketers — domain experts, not SQL writers.
  • The platform already computes analyses (OOS, product title compliance, …) and exposes them as pre-aggregated datasets scoped to a specific user's filters (client, country, retailer…).
  • The challenge: bridge natural language questions to those datasets — which are dynamic, user-scoped, and multi-table — without shipping the whole schema to the LLM every time.
  • Unlike typical NL-to-SQL tasks on static databases, our tables are virtual: they only exist for a given analysis context and are served via MCP tools.
Differentiator vs. generic NL-to-SQL: We don't query a fixed database — we query ephemeral analysis payloads through MCP tools. The "schema" is discovered at runtime. This shapes every design decision.

2. Domain Context

Key points

  • Two main analysis types in scope (at time of writing):
    • OOS (Out-of-Stock) analysis: metrics like OOS rate, variation, competitor index, revenue impact.
    • Product Title Compliance: compliance_rate, health_score, fail-to-pass transitions.
  • Each analysis is scoped to a user, a retailer scope, a time window — not a global table.
  • Multiple tabs per page → multiple analysis contexts. We introduced AnalysisEntry to carry {analysis_type, tab, analysis_id} so the agent can route correctly.
  • Analysis types have distinct metric vocabularies (e.g. "index" means something different on OOS vs. Product Title) → the same question requires different interpretation depending on context.

3. System Architecture — Multi-Agent Design

Overall flow

User Message→ Supervisor Agent→ Route decision→ RAG / Alerts / Backend / Insight→ Verification→ Response

Agent roles

AgentResponsibilityModel
SupervisorIntent routing, conversation state, approval flowsGemini 3 Flash
RAGDocumentation / knowledge base Q&AGemini 3 Flash
AlertsThreshold-based alert management via MCPGemini 3 Flash
BackendStructured data API calls (filters, entities…)Gemini 3 Flash
InsightNL → SQL → analysis results synthesisGemini 3.1 Pro (planner/synthesizer) + Flash (executor)
VerificationCross-checks answers, validates entity IDsGemini 3 Pro Preview
Design choice: Supervisor is cheap (Flash) — it only routes. Insight is expensive where it matters (planning + synthesis) and cheap where it doesn't (SQL execution). Verification sits outside the hot path and scores in MLflow traces.

MCP as the integration layer

  • All data access goes through MCP (Model Context Protocol) tools — not direct DB connections.
  • MCP client is cached per user; per-request headers (auth, session, agent type) are injected at call time via an interceptor — no token leakage across sessions.
  • This decouples the chatbot backend from data infrastructure — the MCP server owns the SQL, the agent owns the reasoning.

4. The Insight Agent — Core Idea

What it does

  • Takes a natural language question + a set of AnalysisEntry objects (analysis_type + analysis_id).
  • Discovers available tables for that analysis at runtime.
  • Plans a multi-step query strategy.
  • Executes SQL (or pre-built shortcut tools) for each step, in parallel.
  • Synthesizes a coherent, domain-aware narrative answer.

Why not a simple ReAct loop?

Problem with pure ReAct (tool-calling loop):
  • Unpredictable number of tool calls — latency spikes.
  • LLM may call get_table_details for every table (20+) before writing SQL — slow and costly.
  • Hard to enforce analytical best practices (always analyze multiple dimensions, default top_n=5…).
  • Difficult to parallelize steps.
  • Prior stateful implementation (with SummarizationMiddleware, Postgres checkpointer) added complexity that wasn't needed for a stateless sub-agent called by the supervisor.

5. Evolution of the Agent — Three Approaches

Approach 1 — Basic LLM Tool-Calling Agent (v0)

  • Insight agent as a standard LangGraph ReAct agent with MCP tools.
  • Had access to: get_available_tables, get_table_details, query_tables.
  • Stateful: used AsyncPostgresSaver checkpointer + SummarizationMiddleware.
  • Model: injected system prompt with analysis context and user context.
Problems observed:
  • Called get_table_details on all tables → sometimes 20 API calls before any SQL.
  • Inconsistent analytical depth: sometimes analyzed 1 dimension, sometimes 5.
  • No structure to the plan — hard to parallelize or evaluate step-by-step.
  • Session state added overhead for a sub-agent that doesn't need it (supervisor owns the conversation state).
  • System prompt alone was not enough to reliably instill domain-specific analytical patterns.

Approach 2 — Plan+Execute with StateGraph (v1)

  • Replaced the ReAct loop with a deterministic 5-node StateGraph (LangGraph).
  • Separation of concerns: discovery is deterministic, planning is expensive LLM, execution is cheap LLM.
discover_tables→ plan→ discover_details→ execute→ synthesize
  • discover_tables: One deterministic MCP call → lightweight catalog [{table_name, description}]. No LLM involved.
  • plan: Expensive LLM (Pro) produces ExecutionPlan — structured output with list of PlanStep objects. Each step specifies tables, expected output, reasoning.
  • discover_details: Fetch full column schemas only for tables the plan selected. Parallel, capped at 3 concurrent calls.
  • execute: Cheap LLM (Flash) generates SQL per step; steps run in parallel (semaphore: max 3 concurrent).
  • synthesize: Expensive LLM (Pro) turns execution results into a narrative answer with domain-aware framing.
Gains: Predictable number of table-discovery calls. Parallelism at the right points. Structured plan enables evaluation of individual steps. Agent is now stateless (no checkpointer needed).

Approach 3 — Plan+Execute + Shortcut Tools (v2, current)

  • Identified recurring SQL patterns that the LLM kept re-generating with slight variations.
  • Extracted them into pre-built shortcut tools — Python code wrapping correct SQL, filters, sort orders and limit semantics.
  • Planner LLM now sees the shortcut tool catalog alongside raw tables. It can set tool_name on a step instead of relevant_tables.
  • Executor dispatches to the shortcut tool directly — no SQL generation needed for those steps.
Gains: Correct SQL for common patterns (no hallucinated column names). Faster (no SQL generation step). Deterministic output for high-frequency queries. Domain knowledge lives in code, not prompt.

6. Plan+Execute Deep-Dive

Structured output for the plan

class PlanStep(BaseModel):
    tool_name: Optional[str]        # shortcut tool if applicable
    step_description: str           # "what this step computes"
    relevant_tables: List[str]      # namespaced table names
    expected_output: str            # "a single number", "top 5 rows"

class ExecutionPlan(BaseModel):
    steps: List[PlanStep]
    reasoning: str                  # why this plan answers the question
Using with_structured_output(ExecutionPlan) on Gemini Pro — no JSON parsing hacks. Reliable, deterministic format. The structured plan is also what MLflow traces capture for evaluation.

Lazy table schema discovery

  • Phase 1 (always): get_available_tables → catalog of names + descriptions. Typically 10–25 tables. No schemas yet.
  • Phase 2 (after planning): get_table_details only for tables the plan selected. From 25 → typically 1–5 calls.
  • This dramatically reduces MCP API calls and token usage vs. fetching all schemas upfront.

Concurrency model

semaphore = asyncio.Semaphore(3)  # MAX_CONCURRENT_STEPS

results = await asyncio.gather(
    *[execute_step(step) for step in plan.steps],
    return_exceptions=True,
)
  • Steps are independent (each queries different tables) → safe to parallelize.
  • Semaphore prevents overwhelming the MCP server.
  • Table-detail fetching is also parallelized with the same pattern.

Two-model strategy

executeGemini 3 Flash (executor)Mechanical: given schema + description, write SQL. Fast + cheap.
synthesizeGemini 3.1 Pro PreviewNeeds domain knowledge to frame the answer correctly

7. Shortcut Tools — Encoding Domain Knowledge

The LLM kept generating the same SQL patterns — but not always correctly. Column names were sometimes hallucinated, sort orders inverted, limit semantics wrong. The fix: encode the correct SQL as Python tools. The LLM selects which tool to use; humans write the SQL.

Examples

ToolAnalysis typeWhat it computes
get_oos_macro_kpisOOSGlobal OOS rate, variation, competitor average, product count
get_oos_dimension_rankingOOSTop/bottom N by OOS across retailer/brand/category/subcategory
get_oos_worst_productsOOSTop N products with worst OOS, with product identifiers, brand, subcategory
get_pt_macro_kpisProduct TitleGlobal compliance rate, health score, pass/fail transitions
get_pt_dimension_rankingProduct TitleTop/bottom N by compliance across dimensions
get_pt_worst_productsProduct TitleLeast compliant products with retailer vs. client title

Analysis-type-scoped visibility

def create_insight_tools(query_tool, analysis_type_ids=None):
    """Return only the tools relevant to the current analysis type(s)."""
    all_tools = [GetOosMacroKpis(...), GetPtMacroKpis(...), ...]
    if analysis_type_ids is None:
        return all_tools
    return [
        t for t in all_tools
        if any(aid in t.metadata["analysis_types"] for aid in analysis_type_ids)
    ]
  • The planner prompt only lists tools relevant to the current analysis type — no cross-contamination.
  • New analysis types get new shortcut tools; existing tools are unaffected.

8. Analysis-Specific Guidance from MLflow Prompts

The problem

  • For a generic "overview" question on OOS, the correct behavior is: macro KPIs → retailer breakdown → category → subcategory. On Product Title, the pattern is slightly different. A generic prompt can't know this.
  • Hardcoding patterns in the codebase couples business logic and code.

The solution: convention-based prompt registry

# Convention: {assistant}-guidance-{analysis_type}-prompt@{env_tag}
# Falls back to: {assistant}-guidance-default-prompt@{env_tag}

def _load_analysis_guidance(self, analysis_type: str) -> str:
    env_tag = settings.insight_guidance_tag   # e.g. "staging" or "prod"
    try:
        prompt = self.load_prompt(f"guidance-{analysis_type}", env_tag)
    except Exception:
        prompt = self.load_prompt("guidance-default", env_tag)
    return prompt.to_single_brace_format()
  • Guidance prompts live in MLflow Prompt Registry — versionable, A/B-testable, no code deploy needed to iterate.
  • Tagged per environment (e.g. staging, prod) so different environments use different versions.
  • Cached in-memory per analysis type for the lifetime of the agent instance.
  • Example guidance includes: which tools to prefer for each question type, metric semantic explanations, default values (top_n=5), competitor comparison patterns.
Impact: Product / data teams can tune analytical behavior without touching code. Evaluation scores improved significantly on "overview" and "extreme values" query types after adding analysis-type-specific guidance.

9. Challenges & Solutions

Challenge 1 — Multi-Tab Analysis IDs

Problem: Dashboard pages have multiple tabs. Each tab generates its own analysis with a different analysis_id. A single query might need data from tabs 1 and 2.

Solution: Introduced AnalysisEntry = {analysis_type, tab, analysis_id}. All MCP calls receive a list of analysis IDs. The supervisor accumulates entries across multiple run_analysis calls. The insight agent passes all IDs to every MCP tool call. Each response carries the full list so the frontend can route results to the correct tab.

class AnalysisEntry(BaseModel):
    analysis_type: str
    tab: str
    analysis_id: str
Challenge 2 — Entity Resolution for Context Tools

Problem: Users say "Carrefour", the system needs an internal entity ID. Passing the full entity directory in every prompt is too expensive (tokens). Relying on the LLM to guess IDs is unreliable.

Solution: EntityIndex with fuzzy matching via rapidfuzz: prefix boosting, exact-token bonus, high-confidence threshold. The LLM calls search_entities(query="Carrefour") → gets the correct entity ID back. Falls back to returning all entities when no good match is found, letting the LLM do semantic matching on a small set.

Challenge 3 — Cross-Page Analysis

Problem: Users sometimes ask about a different analysis type than the one they're viewing ("run product titles analysis while I'm on the OOS page").

Solution: Analysis-triggering tools now accept a target_analysis_type parameter. A resolver helper automatically carries over current-page filters to the target context. The sub-agent can then access those filters when building its queries.

Challenge 4 — Stateful vs Stateless Sub-Agent

Problem: Early implementation used AsyncPostgresSaver + SummarizationMiddleware in the insight agent — same as the supervisor. This was unnecessary complexity and caused checkpoint namespace collisions.

Solution: The insight agent is now stateless. Conversation memory lives in the supervisor's checkpoint. The insight agent receives its full context in the request and returns a self-contained result. Much simpler and easier to test.

10. Evaluation Framework

Setup

  • Eval queries are stored as JSON files per agent type, organised by analysis domain.
  • Each query has: user_message, expected_trajectory (tool call sequence), expected_answer.
  • Runs are tracked in MLflow with experiment IDs; scores are fetched via a custom reporting script.
  • Evaluation job is CI-triggered (GitHub Actions) for regression detection.

Metrics tracked

MetricWhat it measures
Trajectory matchDid the agent call the expected tools in the right order?
Answer accuracyLLM judge comparing generated answer to expected answer
LatencyEnd-to-end response time per query
Tool call countEfficiency — fewer calls = better (for same accuracy)
Key insight from evaluation: Trajectory match is a leading indicator. If the agent calls the wrong tables or skips a dimension, the answer will be wrong even if it sounds plausible. Tracking the tool call sequence, not just the final answer, is essential.

Model evolution driven by evals

  • Upgrading the insight planner to a stronger reasoning model produced meaningful accuracy improvements on complex multi-step questions.
  • Moving the verification agent to per-agent scoring enabled finer-grained regression detection.
  • Shortcut tools adoption tracked via eval: % of steps using a shortcut vs. raw SQL generation.

Notable observations:
  • v0 variance is high — trajectory swings 20–50% across seeds vs. 60–80% for v2. The ReAct loop is less deterministic; a different query distribution changes how the model chooses tool call order.
  • Most of the trajectory gain comes from v0→v1 (+23 pp) not v1→v2 (+7 pp) — the structured graph alone constrains tool call order. Shortcuts contribute primarily to quality and latency, not trajectory adherence.
  • v1 latency is the worst (~78s) — Plan+Execute adds orchestration overhead without the shortcut speed benefit yet. v2 recoups this by skipping SQL generation for common patterns.
  • Insight correctness is similar across all versions (0.880–0.930) — when the agent retrieves the right data, it phrases answers correctly regardless of approach. The gap is in what data gets retrieved (completeness), not how it is interpreted.
  • Seed C is the hardest — v0 trajectory drops to 20%, quality to 0.783. The hardest queries stress-test the ReAct loop most, making it the most informative seed for regression detection.

Key lessons

  1. Structure beats prompting for multi-step reasoning. A StateGraph with explicit nodes is easier to debug, evaluate, and optimize than an open-ended ReAct loop. Each node has a single clear job.
  2. Separate planning and execution; use different model sizes. The planner needs the best reasoning model. The SQL writer just needs a fast model with the schema. This cut latency significantly without accuracy loss.
  3. Encode recurring patterns as code, not prompt. Shortcut tools are more reliable than asking the LLM to regenerate the same SQL every time. They also make errors obvious (wrong SQL → fix the tool, not the prompt).
  4. Domain guidance decoupled from code is a superpower. A prompt registry (we used MLflow) lets domain experts tune analytical behavior without code deploys. Version control + environment tagging gives safe promotion paths.
  5. Lazy schema discovery scales better than eager fetching. Fetching details only for plan-selected tables (vs. all tables) reduced MCP calls by ~75% on typical queries.
  6. The sub-agent should be stateless. Let the orchestrator (supervisor) own conversation state. Sub-agents are simpler, more testable, and easier to replace. Don't add checkpointers where you don't need them.
  7. Trajectory evaluation catches regressions that answer scoring misses. An answer can sound correct while querying the wrong dimension. Tracking tool call sequences is as important as evaluating the final response.

12. Future Work

  • More shortcut tools. Cover more common analytical patterns (trend analysis, revenue impact, segment comparisons) to reduce raw SQL generation to an exception path.
  • Plan validation node. Add a lightweight LLM check between plan and execution to catch plans that reference non-existent tables or obviously wrong dimensions before any SQL runs.
  • Adaptive parallelism. Dynamically set the semaphore limit based on measured MCP server capacity, instead of a hardcoded constant.
  • Per-step result caching. Cache shortcut tool results within a session when the same analysis context + args are requested multiple times.
  • More analysis types. Extend to additional analytics modules — each with their own shortcut tools and guidance prompts.
  • User feedback loop → prompt tuning. Feed user satisfaction scores (thumbs up/down) back into prompt iteration cycles via the prompt registry.
