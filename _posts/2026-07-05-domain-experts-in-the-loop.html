There is a comfortable story about AI systems that goes like this: engineers build the machine, the model provides the intelligence, and quality is an emergent property of good code and a big enough model. It is a tidy story. It is also wrong in the one place that matters most — the place where the system decides what "a good answer" even means.

That place is the harness. And the harness is where domain experts belong. This post is about what that collaboration actually looks like from every side — including the sides nobody puts on a slide, because they are the awkward ones.

1. What "the harness" actually is

When people evaluate an "AI agent," they usually measure one number: did it get the task right? But an agent is not one thing. A recent survey on evaluating LLM agents makes the split explicit, and it is the most useful frame I know for this whole topic:

backbone LLM + harness = agent

The backbone LLM is the model's raw capability — its knowledge, its reasoning, what it learned in training. You don't edit it; you select it. The harness is everything wrapped around it: the prompting strategy, the tool interfaces, the memory and state, the routing logic, the retrieval, and the evaluation criteria. The survey's core argument is that end-to-end scores conflate these two, and until you decouple them you cannot tell whether a failure came from the model not knowing something or from the harness orchestrating it badly.

Here is the punchline that makes domain experts unavoidable: you cannot fine-tune a manufacturer's definition of "a critical out-of-stock rate" into a model's weights, and you wouldn't want to. That knowledge is volatile, client-specific, and contested. It belongs in the harness — in prompts, in tool design, in the eval set. And the harness is authored, not trained. Someone has to write down what "correct" means. That someone is a domain expert.

So the real question is not whether experts belong in the loop. It is which loop, doing what, and at what cost to everyone involved.

2. Where expertise enters the system

Whatever the domain, expert knowledge enters an agent through a small number of recurring doors. The category labels below are generic; the parenthetical examples come from one system I've worked on — an e-commerce chatbot: a retail-analytics assistant for FMCG brands, embedded in a digital-shelf dashboard, whose users are category managers monitoring how their products sell across online retailers and asking things like "which SKU drove the OOS spike last week?" — but the categories transfer to a legal, clinical, or operations agent unchanged. The four doors are not equal.

Door (generic)What the expert authorsHalf-lifeWho else can do it
Success criteriaThe ordered reasoning steps or rubric a correct answer must satisfy — not just the question and the answer (e.g. expected trajectories fed to an LLM judge)LongNobody. This is the purest expert artifact.
Behavioral guidanceDomain heuristics injected before the model acts: thresholds, defaults, what "critical" means (e.g. per-page guidance prompts: "below 70% is critical")MediumAn engineer can format it; only an expert knows the values.
Domain-optimized pathsThe decision that a frequent request deserves a hand-built, cheaper, more reliable path instead of free-form generation (e.g. shortcut tools that bypass generated SQL)LongCo-authored: expert names the pattern, engineer builds it.
Input & intent resolutionMapping messy real-world input to canonical meaning: matching misspelled or colloquial names to real entities, normalizing units and date ranges, disambiguating "last week" or "my top brands" (e.g. fuzzy entity matching plus canonicalizing vague requests)ShortPartly automatable (fuzzy match, parsers), but the ground truth is domain.

Notice the "half-life" column. Input mappings churn constantly; a success criterion that encodes how a good analyst reasons about a problem stays true for years. This is not a footnote — it determines how much of an expert's scarce time is worth spending on each door. The highest-leverage expert artifact is usually the one with the longest half-life and the fewest substitutes: the success criterion.

An eval query that ships with an expert-authored expected trajectory turns the LLM-as-judge from a vibes machine into a checkable instrument. The judge is handed the ideal reasoning path and asked whether the actual run followed it. The evaluation is only ever as good as that trajectory — which means the entire quality signal of the system rests on an artifact that only a domain expert can produce. That is real power, and, as we'll see, real burden.

One honest caveat to "nobody else can do it." Expected trajectories are one shape of success criterion; the other is the rubric — a per-case checklist of criteria a good answer must satisfy, each independently checkable. The largest real-world validation of expert-authored evaluation is rubric-shaped: HealthBench had 262 physicians write 48,562 conversation-specific criteria, then let a model-based grader check them at scale. And the expert monopoly is softening: a study across 823 clinical encounters found LLM-generated rubrics reaching clinician-LLM agreement on par with clinician-clinician agreement — at roughly a thousandth of the cost. The lesson is not that experts are replaceable; it's that their highest-leverage role shifts one level up: from writing every criterion to validating and calibrating the machinery that writes them. Which is the amortization argument of Section 5, taken to its logical end. Either way, calibrate before you trust: measure judge-vs-expert agreement on a held-out slice first, and treat the judge's score as meaningful only after that number is known.

3. Beyond one system: the fuller map

Those four doors are the ones this particular system leans on. They are not the whole building. Plenty of production agents route domain expertise through doors that a retail-analytics assistant simply never opens — and if you're designing a system in a higher-stakes field, some of these stop being optional. Here is the wider catalogue, with an honest note on whether the example system uses each.

Touch pointWhat the expert decidesIn the example system?
Guardrail & policy authoringWhat is out of bounds, and the refusal / redaction / disclaimer rules — the domain-specific "you may not say or do this." Increasingly authored as user-defined policies a guardian model enforces, rather than baked into one prompt.Partial — there's a human-approval gate for destructive actions, but no expert-authored policy layer.
Adversarial probingThe domain-specific ways the system fails that a generic red-team never imagines — the trick question, the misleading-but-plausible query, the compliance landmine. Experts supply attacks, not just labels.No — evaluation is cooperative (real query patterns), not adversarial.
Authoritative-source curationFor retrieval systems: which documents are ground truth, how conflicts between sources resolve, and what the system should refuse to answer from. The corpus is an expert artifact.Partial — a docs corpus exists, but source authority and conflict rules aren't expert-curated.
Preference / reward dataWhen you fine-tune or train a reward model, experts are the preference labelers. Off-the-shelf reward models are known to struggle in specialized domains precisely because they lack this data.No — the system selects models, it doesn't train them.
Action-affordance designWhich actions the agent is permitted to take at all, and each action's side-effect risk tier — the difference between "read a number" and "delete a live alert."Yes, implicitly — the tool set and its approval gates encode this, though it's rarely framed as an expert decision.
Abstention & escalationWhen the agent must say "I don't know," hand off to a human, or refuse to guess. In many fields a confident wrong answer is far costlier than an "I can't answer that."No — there's no explicit expert-set abstention threshold.
Scenario / simulated-user designRealistic evaluation scenarios and simulated users for dynamic eval — expert-designed situations that probe judgment, not just fixed question/answer pairs.Partial — eval runs against a demo user experts configure for corner cases (locale, feature flags like report generation, per-KPI/page context, client-specific rules and settings). Parametrized fixtures, not yet an autonomous adaptive user.
The point of this table is not that the example system is deficient — it made reasonable choices for a cooperative, low-stakes analytics setting. The point is that "domain expert in the loop" means radically different work depending on the stakes. A clinical or legal agent that skipped guardrail authoring, adversarial probing, and abstention thresholds would be negligent; an internal analytics helper that added all three would be over-engineered. Part of the expert's first job is deciding which doors even need to exist.

4. All sides: four viewpoints on one loop

"Expert in the loop" gets drawn as a friendly circle with a person in it. The reality is four parties with different incentives, and the friction between them is the topic. A 12-week ethnographic study of a team building a pedagogical chatbot found that the hard parts were almost never technical — they were about motivation, trust, and who owns knowledge. Here are the four seats at the table.

Side 1 — The domain expert

  • What they want: the system to stop being confidently wrong about their field. They can spot a bad answer in half a second and struggle to explain why in a way a machine can consume.
  • What it costs them: this is not their day job. The ethnographic study flags expert motivation and trust as a first-order challenge — experts disengage when their input vanishes into a black box and nothing visibly improves.
  • The trap: "just write down what good looks like" sounds easy and is genuinely hard. Tacit expertise resists being turned into an ordered checklist. The first three trajectories are exhausting; the expert's instinct is richer than any list they can produce on demand.

Side 2 — The engineer

  • What they want: expert knowledge in a form the harness can execute — a JSON file, a prompt tag, a tool schema. Structured, versioned, testable.
  • What it costs them: building the participatory scaffolding — templates, review UIs, deploy paths — so a non-engineer can contribute without filing a PR. The study calls out participatory design barriers as a recurring failure: teams underinvest in the machinery that lets experts actually participate.
  • The trap: the engineer becomes a bottleneck and a translator, subtly reshaping expert intent into whatever is easy to code. Every hop from "what the expert meant" to "what the harness does" leaks meaning.

Side 3 — The system itself (LLM-as-judge)

  • What it "wants": unambiguous criteria. It will happily score a response 0.91 against a trajectory that encodes a subtly wrong idea of quality. The judge is faithful, not wise.
  • What it costs everyone: the machine launders the expert's assumptions into numbers. A bias in the trajectory becomes a metric, a metric becomes a target, and now the whole system optimizes toward one expert's opinion — dressed up as objective measurement.
  • The upside: done well, the survey notes, humans and judges cover different ground — the judge scales across hundreds of retailers and thousands of brands while the expert supplies the qualitative dimensions (tool appropriateness, reasoning coherence, task alignment) that automated metrics miss. The JADE line of work on expert-grounded dynamic evaluation pushes exactly this: keep the expert's judgment in the scoring loop rather than freezing it into a static rubric.

Side 4 — The product / business

  • What they want: the system to improve without an engineering sprint per fix. If a KPI owner can update a guidance prompt and see quality rise on the next request, the improvement loop is decoupled from the release cycle. That is the whole prize.
  • What it costs them: owning the process. "Who are the right experts, and is that ownership real or aspirational?" is the question that quietly sinks these programs. Expert time is a budget line, and nobody put it in the plan.
  • The tension: the business wants autonomy for experts and guardrails against a well-meaning expert shipping a bad prompt to prod. Those pull in opposite directions.
The single most under-discussed conflict: the expert and the judge can both be wrong in the same direction. If the expert's trajectory reflects a common-but-flawed analytical habit, the eval will reward the agent for reproducing it, production feedback may even confirm it (users share the habit), and every signal in your quality loop agrees — while the system is quietly wrong. Consensus is not correctness. This is the failure mode that no amount of harness engineering catches, because it lives in the ground truth itself.

5. The spectrum of involvement

"Expert in the loop" is not one arrangement — it's a spectrum of increasing leverage and increasing burden. The ethnographic study observed teams sliding along it as expert availability shifted, and named the two endpoints worth aiming for: co-developed evaluation criteria and hybrid evaluation that blends expert, developer, and model judgment.

ModeExpert roleLeverageBurden on expert
AnnotatorLabels outputs good/bad after the factLow — reactive, per-itemLow but endless
AuthorWrites guidance prompts / thresholds directlyMedium — shapes behaviorMedium
Judge-designerAuthors expected trajectories the LLM-judge scores againstHigh — defines "correct" at scaleHigh, front-loaded
Co-developerBuilds the eval criteria with engineers, iteratingHigh — shared ownershipHigh, ongoing
Hybrid loopExpert + developer + model each score; disagreements surfaceHighest — catches all three's blind spotsDistributed, sustainable

The move that pays off is climbing from annotator (where the expert is a bottomless well of labels and burns out) to judge-designer and hybrid (where the expert's effort is amortized — one good trajectory scores thousands of runs). The annotator mode feels productive and is a trap; it never compounds.

Adjacent research on benchmarking the actual value of human-in-the-loop collaboration and human-in-the-loop software agents lands on the same lesson from the coding-agent world: the win is not "human reviews everything" but "human is inserted at the few steps where their judgment is decisive." Ubiquitous oversight doesn't scale; targeted oversight does.

6. Failure modes nobody advertises

Every writeup celebrates the loop closing. Here are the ways it quietly doesn't.

  • Expert scarcity dressed as a data problem. The real constraint is rarely the pipeline; it's that the two people who understand the domain have day jobs. The study documented teams inventing workaround data collection and augmentation strategies precisely because expert time ran out — useful, but a signal that the loop is under-resourced, not that it's solved.
  • Overfitting to one expert's taste. An eval set is a portrait of whoever wrote it. Two senior analysts routinely disagree about what a good OOS answer emphasizes. If your trajectories come from one person, you have encoded their preferences as ground truth — and you'll never see it, because your metrics agree with your rubric by construction.
  • Stale guidance. A guidance prompt authored when a threshold was 70% keeps enforcing 70% long after the business moved to 65%. Guidance has a half-life; without a review cadence it rots into confidently-outdated advice — worse than no guidance, because it looks authoritative.
  • Knowledge ownership disputes. The study names this directly: who owns the expert knowledge once it's in the system? When the expert leaves, does their trajectory set stay? Is it "the company's eval" or "Maria's opinions"? Unowned artifacts decay.
  • Consent and attribution. Experts contribute knowledge that becomes a durable asset. The study's recommendation for transparent consent frameworks is not legal boilerplate — it's what keeps experts willing to contribute a second time.
  • The evolving-role blindspot. An expert's job in month one (bootstrap the eval set) is not their job in month six (adjudicate the disagreements the hybrid loop surfaces). Teams that treat the role as static lose the expert right when their input gets most valuable.
Meta-failure: treating the expert loop as a phase instead of a fixture. "We did the expert workshop" is the tell. The domain moves, the client base changes, the model gets swapped — and a quality definition frozen at launch is a liability accruing interest. The loop is infrastructure, not a milestone.

7. A durable operating model

Pulling the sides together, here is what a loop that actually survives contact with reality looks like. None of this is exotic; the discipline is in keeping it running.

  1. Make the artifact the unit of work, and version it. Guidance prompts live in a registry with tags and rollback (dev → staging → prod), decoupled from code deploys. Trajectories live in reviewed JSON. If expert knowledge isn't a diffable, ownable artifact, it isn't in the loop — it's in someone's head.
  2. Climb the spectrum deliberately. Start experts as annotators only long enough to bootstrap; move them to trajectory-authoring fast, because that's the effort that compounds. Track the ratio of amortized work (trajectories) to per-item work (labels) — if it's not shifting toward the former, the loop is burning the expert.
  3. Close the visible feedback gap. The number-one killer of expert motivation is contributing into a void. When an expert ships a guidance prompt, the trace on that KPI page should show whether quality moved. Show them the delta. That's what earns the next contribution.
  4. Build the hybrid triangulation for real. Expert judgment, LLM-judge scores, and production user feedback should be three independent signals. When they disagree, that disagreement is your highest-value data — it's the only thing that catches the "expert and judge wrong together" failure. Route disagreements to a human, on purpose — and calibrate the judge against the expert on a held-out slice before the triangle means anything.
  5. Schedule the rot check. Guidance and trajectories get a review cadence, an owner, and an expiry prompt. "Is this threshold still right?" is a calendar event, not a hope.
  6. Resource it honestly. Put expert hours in the plan as a line item. A loop that depends on borrowed goodwill runs exactly until the expert gets busy.
The thing worth internalizing: the harness is the only layer where you can encode "what this specific business means by correct" — and that layer is authored by hand. So the quality ceiling of the whole system is set not by the model you picked, but by how well you run the collaboration with the people who hold the definition of good. That is a people-and-process problem wearing an engineering costume.
